Basic Design

筋トレ日記アプリ 基本設計書

要件定義書をもとに、Web/PWA 前提の画面構成、データモデル、IndexedDB 保存方針、主要なデータフローを整理する。 本書は概念設計を中心とし、後続の詳細設計・実装設計へつなげるための土台とする。

1. 概要

目的

日々の筋トレ内容と体重をスマホで素早く記録し、種目ごとの最大重量推移と体重推移を確認できるようにする。

MVP範囲

今日の記録、日付ごとの筋トレ記録、体重記録、マイ種目管理、プリセット追加、メニューテンプレート、グラフ、端末内保存を対象とする。

主要価値

  • 今日の記録を迷わず残せる
  • 種目やメニューを事前準備できる
  • 記録結果がグラフで返ってくる
  • ログインなし
  • クラウド同期なし
  • CSVエクスポートなし

2. システム構成

MVP は Web/PWA として構成する。スマホでの片手操作を主対象とし、デスクトップ表示は補助的に扱う。

ユーザー スマホ中心に記録・確認する
→
Web/PWA アプリ 画面、入力、自動保存、集計、グラフ表示
→
IndexedDB 端末内に設定・種目・記録・メニューを保存
保存データはエンティティごとに分離する。最大重量や期間内変化量などの派生値は保存せず、表示時に保存済みデータから集計する。

5. 機能設計

今日の記録

  • 初期表示は当日の日付とする。
  • 日付ごとに体重と複数の種目カードを入力できる。
  • 各種目カードには複数セットを追加できる。
  • 入力内容は自動保存する。

種目管理・プリセット

  • マイ種目を自由に追加・編集できる。
  • 使わない種目は削除ではなく非表示にする。
  • プリセットから複数種目を選択してマイ種目へ追加できる。

メニューテンプレート

  • 1回分の筋トレメニューを作成できる。
  • メニュー内の種目は並び替えできる。
  • 今日の記録へ複数種目をまとめて追加できる。

グラフ

  • 重量推移と体重推移をタブで切り替える。
  • 期間は1か月、3か月、6か月、全期間から選べる。
  • 初期期間は3か月とする。

6. データモデル

共通方針

  • ID は UUID 文字列
  • 日付は YYYY-MM-DD
  • 日時は ISO 8601
  • 重量・体重は kg
  • 空セットは null 保持

主要エンティティ

UserSettings 端末内に保存する設定
  • weightUnit
  • defaultGraphRange
  • isSetupCompleted
ExercisePreset アプリ標準の種目候補
  • name
  • bodyPart
  • equipmentType
Exercise ユーザーのマイ種目
  • sourcePresetId
  • isActive
  • sortOrder
WorkoutDay 日付単位の記録
  • date
  • memo
WorkoutExercise 特定日の種目カード
  • workoutDayId
  • exerciseId
  • sortOrder
WorkoutSet 種目カード内の1セット
  • workoutExerciseId
  • weightKg
  • reps
BodyWeightLog 日付ごとの体重
  • date
  • bodyWeightKg
MenuTemplate メニューのひな形
  • name
  • memo
  • sortOrder
MenuTemplateExercise メニューに含まれる種目
  • menuTemplateId
  • exerciseId
  • sortOrder

属性一覧

エンティティ 主な属性 説明
UserSettings id, weightUnit, defaultGraphRange, isSetupCompleted, createdAt, updatedAt 端末内に保存するユーザー設定。
ExercisePreset id, name, bodyPart, equipmentType, sortOrder アプリ標準の種目候補。ユーザー編集対象ではない。

9. 業務ルール

項目 ルール
最大重量同一日・同一種目の全セットの中で、一番重い重量をその日の最大重量とする。
同じ種目の追加今日すでに同じ種目がある場合は、新しい種目カードを作らず既存カードに新しいセットを追加する。
セット追加既存セットがある場合は前セットの重量・回数をコピーする。セットがない場合は空の1セット目を作る。
体重記録トレーニング記録とは独立して日付ごとに保存する。トレーニングしない日も記録できる。
プリセット追加プリセットは直接今日の記録へ出さず、マイ種目に追加された後に種目選択シートへ表示する。
メニュー追加メニューを今日に追加した場合、含まれる種目カードと空の1セット目を生成する。
グラフ初期表示重量推移・体重推移ともに初期期間は直近3か月とする。
単位重量と体重は kg を基本とする。

10. 非機能設計

ユーザビリティ

  • スマホ片手操作を前提とする。
  • 日々の入力は画面遷移を少なくする。
  • 自動保存を基本にする。

パフォーマンス

  • 今日の記録画面は即時に表示する。
  • 検索と絞り込みは入力に追従する。
  • グラフ集計は通常利用の記録量で快適に表示する。

データ保護

  • MVP では端末内保存とする。
  • 削除操作には確認を入れる。
  • 将来の同期に備えてデータ構造を分離する。

保守性

  • プリセット、マイ種目、記録、メニューを別エンティティとして扱う。
  • グラフ用集計は保存データから導出する。
  • 推定1RMや総ボリュームを追加しやすい構造にする。

11. テスト・受け入れ基準

番号 基準 確認観点
A-01今日の記録画面で体重を入力できる。入力後、同じ日付を再表示しても体重が残っている。
A-02今日の記録画面で種目を追加できる。マイ種目から選択した種目カードが日付画面に表示される。
A-03セット単位で重量・回数を記録できる。複数セットが保存され、編集後も保持される。
A-04プリセットからマイ種目を追加できる。追加した種目が種目選択シートに表示される。
A-05メニューテンプレートから今日の記録へ複数種目を追加できる。メニュー内の種目カードが並び順通りに追加され、各種目に空の1セット目がある。
A-06重量推移グラフで日別最大重量を確認できる。同日の複数セットから最大重量だけがグラフに反映される。
A-07体重推移グラフを確認できる。体重記録が日付順に折れ線グラフで表示される。
A-08グラフの初期期間が直近3か月である。重量推移・体重推移ともに初回表示で3か月が選択されている。
A-09記録は端末内に保存される。アプリを閉じて再起動しても保存済みデータが残っている。
A-10スマホ幅で画面が破綻しない。主要画面でテキストやボタンが重ならず、操作可能である。
